# План: что попадает в логи панели и аудит

Документ фиксирует **текущее поведение** журнала событий и аудита в панели, чтобы администратору было понятно, где и какие действия отображаются.

## 1. Три разных «лога» — не путать

| Канал | Где смотреть | Кто пишет | Назначение |
|--------|----------------|-----------|------------|
| **Файл «События»** | Страница `/events` (таблица + CSV по содержимому), путь `ADMIN_EVENTS_LOG_PATH` (часто общий с ботом `data/bot.log`) | В основном **процесс бота** (`RotatingFileHandler`); строки **`[ADMIN]`** дописывает **процесс admin** (`_append_ops_to_events_log`) | Единая лента для админа: цикл бота + явные админ-действия |
| **Файл аудита `[AUDIT]`** | На диске: `ADMIN_AUDIT_LOG_PATH` или `data/admin_audit.log` (в UI не показывается) | `admin_main._append_audit_file_line` из `_audit_op` и `_persist_admin_crud_audit` | Строковый журнал операций Docker и CRUD без смешивания с `bot.log` |
| **Stdout / docker logs** контейнера **admin** | `docker compose logs admin` | `uvicorn` + `logging` в `admin_main` (`logger.info` / `warning`) | Отладка, централизованный сбор логов (Loki, ELK и т.д.) |
| **Таблица БД `bot_ops_audit`** | Только БД (опционально, SQL/отчёты); в панели **не** отображается | `admin_main._audit_op`, `_persist_admin_crud_audit` при включённом `ADMIN_AUDIT_CRUD_DB` | Структурированный аудит для запросов к Postgres |

Строки вида `[INFO]`, `[WARNING]` в файле `bot.log` — это **бот**, а не панель. Строки **`[ADMIN]`** — только то, что явно вызвано из **admin** через `_append_ops_to_events_log`.

### 1.1. Назначение каналов с точки зрения эксплуатации

| Канал | Назначение | Кто потребляет | Ретеншн (типично) |
|-------|------------|----------------|-------------------|
| **Файл событий + таблица `/events`** | Оперативный мониторинг: лог бота и `[ADMIN]` в одной ленте | Админ в UI `/events` | При большом файле — хвост **`ADMIN_EVENTS_LOG_SCAN_BYTES`**; ротация бота по **`LOG_MAX_BYTES`** / **`LOG_BACKUP_COUNT`**; в UI только **текущий** файл, не `bot.log.N`. |
| **Файл `[AUDIT]`** | Аудит панели отдельно от `bot.log` | Админ на диске / внешний сбор | Задаётся политикой тома `data/` (ротацию можно настроить logrotate на хосте). |
| **Stdout admin** | Централизованный сбор, алертинг по шаблонам | DevOps, SRE | Задаётся инфраструктурой (Docker logging driver, retention в Loki и т.д.) |
| **`bot_ops_audit`** | Расследование инцидентов, отчётность, compliance | Админ, аудитор, скрипты | Политика на стороне БД: бессрочно, партиции или purge старше N дней |

**Ключевой принцип:**

> **Файл событий** — «посмотреть сейчас» в UI. **Файл `[AUDIT]`** — отдельный журнал действий панели. **БД** — «доказать потом» (SQL). **Stdout** — «собрать везде».

## 2. Текущее состояние: что уже пишется

### 2.1. В файл «Событий» (`[ADMIN]`, процесс admin)

Вызывается только **`_append_ops_to_events_log`** (`admin_main.py`):

| Событие | Текст (смысл) |
|---------|----------------|
| Успешный вход | `Вход в панель login=… ip=…` |
| Выход | `Выход из панели login=…` |
| Docker | `Docker bot/restart scheduled by=…` |
| Docker | `Docker bot/start ok …` / `… stop ok …` (и варианты `failed`) |

**Итого:** сессия + управление контейнером бота. **Нет** записей про пользователей бота (`bot_users`), группы, маршруты, секреты (кроме косвенно через общий файл, если туда пишет бот), настройки `/me/settings`, онбординг, аккаунты панели и т.д.

### 2.2. Stdout процесса admin (`logger`, без гарантии попадания в `bot.log`)

| Событие | Метод / место | Примечание |
|---------|----------------|------------|
| JSON-аудит Docker | `_audit_op` → `logger.info` с `{"level":"AUDIT",…}` | Дублирует смысл `bot_ops_audit` в текстовом виде |
| Обновление секрета (имя + актор) | `secret_updated name=…` | `/secrets`, часть сценариев онбординга |
| Сброс пароля аккаунта панели | `admin_password_reset target=… actor=…` | |
| Смена логина аккаунта панели | `admin_login_changed old=… new=… actor=…` | |
| Ошибки / предупреждения | `logger.warning` / `exception` | Docker, CSRF, Redmine lookup cooldown и т.д. |

Эти сообщения **не** проходят через `_append_ops_to_events_log`, поэтому на странице «События» их **не видно**, если смотреть только хвост файла и нет отдельного сбора stdout.

### 2.3. Таблица `bot_ops_audit` (Postgres)

Заполняется **`_audit_op`** при операциях **Docker** над ботом (start / stop / restart, в т.ч. фоновый restart). При включённом **`ADMIN_AUDIT_CRUD_DB`** туда же пишет **`_persist_admin_crud_audit`** (`action=ADMIN_CRUD`, поля `entity_*`, `details_json`). В веб-панели эта таблица **не показывается** — только запросы к БД и внешние отчёты. В модели SQLAlchemy поле актора называется **`actor_login`**, колонка в БД исторически — **`actor_email`** (`src/database/models.py`).

### 2.3.1. Файл `[AUDIT]` (`ADMIN_AUDIT_LOG_PATH`)

Отдельный текстовый журнал: **`_append_audit_file_line`** вызывается из **`_audit_op`** и **`_persist_admin_crud_audit`** (пока путь файла не отключён). Формат строки: `ДД.ММ.ГГГГ ЧЧ:ММ:СС [AUDIT] …`. Отключение: **`ADMIN_AUDIT_LOG_PATH=-`** (или `none` / `off` / `0` / `false`).

### 2.4. Лог процесса **бота** в тот же файл

Если у бота `LOG_TO_FILE=1` и путь совпадает с `ADMIN_EVENTS_LOG_PATH`, в «Событиях» смешиваются:

- циклы опроса Redmine, предупреждения, `🚀 Бот запущен` и т.д.;
- строки `[ADMIN]` от admin.

Это ожидаемо, но **не заменяет** журнал действий в UI по пользователям/группам.

## 3. Текущее состояние: что **не** логируется явно

Ни **`[ADMIN]`**, ни **`bot_ops_audit`**, ни обязательного структурированного `logger.info` для следующих действий (типичный сценарий «создал пользователя / удалил / группы»):

| Область | Маршруты (примеры) | Заметка |
|---------|-------------------|---------|
| Пользователи бота (Redmine/Matrix) | `POST /users`, `/users/{id}`, удаление, версии | Только изменения в БД |
| Группы | `POST /groups`, редактирование, удаление, маршруты статусов/версий | То же |
| Глобальные маршруты версий | `/routes/version` | То же |
| Устаревшие/редирект маршрутов статусов | `POST /routes/status*` | Если используются |
| Настройки сотрудника | `POST /me/settings` | |
| Первичная настройка | `POST /setup`, `/onboarding/save`, skip, catalog | Секреты в лог не выводить |
| Вход неуспешный | Только ответ 401 в UI | Можно добавить rate-limit + audit без пароля |

Поэтому ожидание «всё, что я делал в панели, видно в Событиях» **сейчас не выполняется** — по задумке в файл попадали в основном **сессия** и **Docker**.

## 4. Цели и принципы расширения

1. **Понятность:** админ видит в «Событиях» значимые действия без шума каждого GET.
2. **Без секретов:** не логировать значения секретов, API-ключи, пароли; маскировать логины по политике `mask_identifier` где уместно.
3. **Единообразие:** один слой (например модуль `src/admin/…` + вызовы из `admin_main`), который опционально пишет в файл, в БД и при необходимости дублирует в stdout.
4. **Производительность:** массовые операции (импорт) — не тысяча синхронных `open/write` на запрос; см. раздел 5 (очередь / агрегация).
5. **Согласованность каналов:** критичные операции (Docker, сброс пароля, смена логина) — предсказуемая запись в БД; для массового CRUD допустима очередь с метрикой глубины и fallback при переполнении.
6. **Совместимость:** не ломать текущий формат `[ADMIN]` для мониторинга; новые типы — с явным префиксом, например `[ADMIN] CRUD …`.

## 5. Риски при внедрении и контрмеры (ревью)

### 5.1. Синхронная запись в файл при массовом импорте

**Риск:** N успешных операций → N синхронных дописей в файл (и при полной схеме — N INSERT в БД) удлиняют HTTP-запрос и могут дать таймаут.

**Контрмеры:**

- **Очередь с фоновым воркером** (`queue.Queue` + поток или отдельный воркер): в обработчике после `commit` только `put_nowait` события; воркер пишет файл и БД. При `queue.Full` — не молча терять: минимум `logger.warning` + строка в stdout (`AUDIT_DROPPED`), опционально счётчик метрик.
- **Агрегация:** для импорта одна строка вида `CRUD bot_user import count=1000 actor=…` вместо 1000 строк.
- Переключатель **`ADMIN_EVENTS_LOG_CRUD`**: можно включать детальный CRUD только там, где нет массовых сценариев.

Набросок API слоя (реализация в `src/admin/`, вызовы из `admin_main` после успешного commit):

```text
log_audit(event) → очередь (обычные CRUD)
log_audit_sync(event) → сразу файл + БД (критичные: Docker, PASSWORD_RESET, LOGIN_CHANGED)
```

### 5.2. Утечка секретов в `details`

**Риск:** в словарь деталей случайно попадут `api_key`, `password`, токены, длинные произвольные строки.

**Контрмеры:**

- Жёсткий **allow-list** полей для типа события **или** санитизация: известный набор ключей (`api_key`, `token`, `secret`, `password`, `key`, …) → маска `***REDACTED***`; длинные строки обрезать (например 100 символов + `…`).
- Реализовать через **Pydantic v2** (`field_validator` на поле `details`) или отдельную функцию `sanitize_audit_details(dict) -> dict`, покрытую тестами.

### 5.3. Рассинхрон: файл записан, БД нет (или наоборот)

**Риск:** падение воркера или БД между записью в файл и INSERT.

**Контрмеры:**

- **Критичные события** — синхронная запись в БД (и при необходимости в файл) в том же запросе, где уже есть сессия; ошибка аудита логируется, политика: fail request vs best-effort — зафиксировать в коде явно.
- Для **очереди CRUD** — мониторинг `audit_queue_size` / счётчики успех/ошибка; при сбое воркера перезапуск процесса admin с потерей необработанной очереди допустим только если приемлемо по политике; иначе — durable outbox (отдельная фаза).

### 5.4. Парсер «Событий»

Сейчас **`src/events_log_display.py`** показывает строки файла как текст (без жёсткого парсера CRUD). Новые строки должны оставаться **однострочными**, без неэкранированных переводов строк в `details`.

### Фаза 2 — Расширение `bot_ops_audit` (оценка 3–4 дня)

**Статус:** реализовано (миграция **`0014_bot_ops_audit_crud`**, поля **`entity_type`**, **`entity_id`**, **`crud_action`**, **`details_json`**; CRUD-панели пишутся с **`action=ADMIN_CRUD`**, **`status=ok`**; Docker-строки по-прежнему **`BOT_RESTART`** и т.д., новые колонки у них `NULL`).

**Переменные:** **`ADMIN_AUDIT_CRUD_DB`** — не задано → как **`ADMIN_EVENTS_LOG_CRUD`**; **`0`** — не писать CRUD в БД; **`1`** — писать в БД (в т.ч. без файлового флага).

**Задачи (оставшиеся):** retry на траниентные ошибки БД; при необходимости индекс по `(actor_email, created_at)`; UI (фаза 3).

**Схема:** отдельная таблица не вводилась: CRUD отличается от Docker значением **`action`** и заполненными опциональными колонками.

### Фаза 3 — Таблица аудита в UI (объединено со «Событиями»)

**Статус:** таблица, фильтры, пагинация и CSV перенесены на **`/events`** (вместе с хвостом файла лога); **`/events/export.csv`** (до 5000 строк, UTF‑8 BOM). Отдельный пункт «Аудит» убран; редиректы **`/audit`** → **`/events`**, **`/audit/export.csv`** → **`/events/export.csv`**.

**Дальше:** доработать UX (календарь, сохранённые фильтры), при необходимости отдельный индекс под типичные отчёты.

### Фаза 4 — Неуспешные входы и безопасность

- Учёт неудачных логинов (без пароля): IP, время, `mask_identifier(login)` → файл и/или audit.
- Согласовать с `ADMIN_LOGINS` и rate limit.

## 9. Метрики (опционально)

В репозитории **пока нет** `prometheus_client`. Если появится потребность в алертах «аудит отваливается»:

- счётчики: `admin_audit_events_total{event_type,status}`;
- гистограмма или summary: время записи в файл/БД;
- gauge: размер очереди аудита.

Подключение метрик — отдельное решение (зависимость, endpoint `/metrics`, согласование с Docker).
