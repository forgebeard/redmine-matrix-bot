# План: глобальный редизайн админ-панели (FastAPI + Jinja2 + HTMX)

## Цели

- Внешний вид в духе референса: **тёмная тема**, **cyan-акцент**, **glass-карточки**, **сайдбар ~288px**, **верхняя полоса** с предупреждением об интеграциях, **топ-бар** с меню пользователя.
- Сохранить текущую **бизнес-логику**, **маршруты**, **формы**, **CSRF**, **сессии**, **Postgres**.
- Упростить сопровождение по сравнению с «одним огромным HTML с табами на JS».

## Решения и обоснования

### 1. Навигация: мультистраничность с общим layout (без SPA-оболочки на первом этапе)

**Решение:** отдельные URL (`/`, `/users`, `/routes/status`, …), все «внутренние» страницы в **одном `base.html`**: сайдбар + топ-бар + `{% block content %}`.

**Почему:**

- Уже есть **разные эндпоинты и редиректы** (логин, онбординг, права). SPA с `hx-get` в `#main-content` ведёт к дублированию маршрутов (полная страница vs фрагмент), к тонкой настройке **истории** и обработке **403/302** на кусок HTML.
- Закладки, «Назад», копирование ссылки работают без доп. кода.
- HTMX можно применять **точечно** (поиск Redmine, фильтры, фрагменты таблиц), не переписывая всё приложение.

**Второй этап (опционально):** «как приложение» с подгрузкой фрагментов — только если появится явная потребность; текущий дизайн этому не противоречит.

### 2. Страница входа: отдельно от shell

**Решение:** `/login`, `/setup`, forgot/reset — **без** сайдбара, в том же визуальном языке (фон, карточка, типографика).

**Почему:** меньше отвлекающих элементов, нет навигации «внутри» до авторизации, проще редиректы и модель угроз.

### 3. Tailwind: не застревать на CDN в проде

**Решение:**

- **Разработка:** Tailwind CDN допустим для скорости прототипа.
- **Продакшен:** **сборка Tailwind** в Docker (`input.css` → минифицированный CSS в образе) **или** эквивалент без Tailwind: **CSS variables + утилитарные классы** под макет.

**Почему:** CDN в проде — внешняя зависимость, сложнее **CSP**, нет purge → тяжелее ответ, риски без сети.

### 4. Иконки: Font Awesome vs SVG

**Решение:** при жёстком CSP или закрытом контуре — **inline SVG** или локальный спрайт; если CDN разрешён — FA6 как в референсе.

**Почему:** те же риски, что у стилей с CDN, плюс контроль размера и лицензирования.

### 5. Шрифт Inter

**Решение:** по возможности **локально** в репозитории или через штатный pipeline; fallback `system-ui`, sans-serif.

**Почему:** предсказуемость в air-gapped средах.

### 6. HTMX: не «везде», а там, где есть выигрыш

**Решение:**

- Оставить **обычные POST** для логина, сохранения форм, удаления с редиректом 303 — надёжно с **CSRF**.
- **hx-get** (и при необходимости hx-post с заголовками) для: поиска Redmine, фильтров, мелких подгрузок.
- Расширять «без перезагрузки» **постепенно**, с `hx-headers` и CSRF из `meta`.

**Почему:** требование «все кнопки на HTMX» часто увеличивает сложность без выгоды; формы уже работают.

### 7. Тёмная тема

**Решение:** сначала **одна тёмная тема** как в референсе; переключатель светлой темы — **отдельная итерация**.

**Почему:** меньше комбинаций при первом крупном редизайне.

### 8. Баннер «Интеграции не настроены»

**Решение:** в топ-баре, кликабельный переход на `/onboarding` (по смыслу как сейчас). Опционально позже: сворачиваемость через `localStorage` (не в БД на первом шаге).

**Почему:** не ломаем серверную логику готовности интеграций.

### 9. Дропдаун пользователя

**Решение:** пункты вести в **реальные маршруты** (например «Настройки» → `/me/settings`; для админа — отдельно согласовать «аккаунты панели» и т.д.), без дублирующих пунктов с одинаковым действием.

**Почему:** меньше путаницы, чем в демо с двумя ссылками на одно и то же.

## Предстарт (ревью перед PR-UI-1)

Сжатые дополнения к плану — снижают риски до начала вёрстки.

| Тема | Действие |
|------|----------|
| **CSP** | Задать политику до продакшена: в dev допускаются внешние CDN (если используются), в prod — `style-src`/`script-src` только `'self'` и нужные inline-хуки для HTMX. Либо отдельный PR только с CSP, чтобы не смешивать с первой вёрсткой. |
| **CSRF + HTMX** | Сервер сейчас сверяет токен из формы с cookie. Для `hx-post`/`hx-delete` добавить приём того же значения в заголовке `X-CSRF-Token` и единый слушатель `htmx:configRequest` в `base.html` (токен в `meta`, значение совпадает с cookie). Поля формы `csrf_token` при необходимости сделать опциональными (`Form(default="")`), если тело запроса не multipart-форма. |
| **Tailwind / CSS** | Либо сборка Tailwind в Docker (отдельный stage Node), либо v1 на **CSS variables + классы** без Node — быстрее первый релиз. |
| **Адаптив** | Breakpoint ~768px: сайдбар скрыт по умолчанию, кнопка «меню», overlay, закрытие по клику вне / Escape. |
| **Длинные `room_id`** | В таблицах: `truncate` + `title="{{ room_id }}"`, моноширинный шрифт. |
| **Пути шаблонов** | Переезд `login` в `templates/auth/` не обязателен — можно оставить `templates/admin/` и не плодить рефакторинг в том же PR, что и дизайн. |
| **Тег / бэкап** | Перед крупным PR: `git tag` на текущий `main`, чтобы откатить визуал без archaeology. |

## Этапы работ

1. **Дизайн-токены и каркас** — новый `base.html`: сетка, сайдбар, топ-бар, слот контента, стили, `meta` для CSRF, HTMX. Публичные страницы (`login`, `setup`, forgot/reset) — `base_auth.html` без shell. *(Статус: **готово** — тёмная тема, сайдбар ~288px, баннер интеграций, дропдаун пользователя, мобильное меню + overlay, CSRF в middleware + `X-CSRF-Token` для HTMX.)*
2. **Миграция существующих страниц** — все шаблоны `templates/admin/*.html` на единый каркас без смены бизнес-логики. *(Статус: **готово** — дашборд, пользователи, маршруты статусов/версий, секреты, аккаунты панели, онбординг, привязка Matrix; общий макрос `macros.html` → `room_cell` для длинных `room_id`; скрытый `csrf_token` на state-changing POST; утилитарные классы в `base.html`.)*
3. **Публичные страницы** — login/setup/forgot/reset: отдельный минимальный layout, тот же визуальный язык. *(Статус: **готово** на `base_auth.html`.)*
4. **Полировка** — адаптив (сайдбар → overlay на узких экранах), hover/focus, доступность дропдауна (Escape, клик вне). *(Частично: overlay, Escape, фокус при открытии/закрытии меню пользователя, стрелки/Home/End, `:focus-visible`; **формы:** единые классы `.form` / `.field` / `.field-hint` / `.choice` / `.inline-range` / `.sub-panel` в `base.html` и `base_auth.html`.)*
5. **Прод-сборка CSS** (если выбран Tailwind build) + при необходимости правки Docker/README.
6. **Тесты и smoke** — обновить `docs/ui-smoke-checklist.md` и при необходимости тесты редиректов/статусов. *(Статус: чеклист синхронизирован с CSRF и таблицами маршрутов.)*

## Риски

- Смешение полных страниц и фрагментов без правила → дублирование шаблонов. **Правило на этапе 1:** полные страницы + точечный HTMX.
- **CSP** и внешние CDN — заранее согласовать или убрать.
- Длинные Matrix `room_id` в таблицах — нужны truncate + `title` или моноширинный блок с переносом.

## Критерии готовности

- Ключевые экраны в новом стиле без регрессий по логину, CSRF, редиректам.
- Сайдбар отражает разделы, активный пункт виден.
- Узкая ширина: контент не ломается, сайдбар не перекрывает критичное без способа открыть меню.
- Документировано, как собирается CSS для продакшена, если не используется CDN.
