# Руководство администратора

Документ для того, кто разворачивает **веб-панель (admin)**, **бота** и **PostgreSQL** из этого репозитория. Конечные пользователи Matrix/Redmine **не вводят** учётные данные базы данных — доступ к БД только у сервисов на сервере через переменные окружения.

См. также: [README.md](../README.md) (общий обзор), [secrets-storage.md](secrets-storage.md) (секреты).

---

## 1. Что входит в стек

| Компонент | Назначение |
|-----------|------------|
| **postgres** | PostgreSQL 16, данные в томе `postgres_data` |
| **admin** | Панель управления (FastAPI), при старте выполняет `alembic upgrade head` |
| **bot** | Процесс опроса Redmine и отправки в Matrix |
| **docker-socket-proxy** | Ограниченный доступ админки к Docker API (перезапуск бота из UI) |

## 2. Развёртывание через Docker Compose (вариант A)

### 2.1. Подготовка

1. Установите [Docker](https://docs.docker.com/get-docker/) и Docker Compose v2.
2. В корне репозитория:

   ```bash
   cp .env.example .env
   ```

3. Для **production** обязательно замените в `.env` как минимум:
   - `POSTGRES_PASSWORD` и при необходимости `POSTGRES_PASSWORD_URLENC` (если в пароле есть символы URL, см. комментарии в `.env.example`);
   - `MATRIX_*`, `REDMINE_API_KEY`;
   - `AUTH_TOKEN_SALT`, `APP_MASTER_KEY` или путь к файлу ключа;
   - список `ADMIN_LOGINS` под логины администраторов панели.

   Для **локальной проверки** можно оставить значения из `.env.example`, но не выкладывайте такой `.env` в git.

4. Создайте каталог для логов (если его ещё нет):

   ```bash
   mkdir -p data
   ```

### 2.2. Запуск

```bash
docker compose up --build -d
```

Если после обновления `docker-compose.yml` появилось предупреждение про **orphan**-контейнер (например, старый одноразовый сервис инициализации пароля), очистка:

```bash
docker compose up -d --remove-orphans
```

### 2.3. Порты (важно для доступа с хоста)

| Куда | Адрес по умолчанию |
|------|---------------------|
| Админка | `http://127.0.0.1:8080` (порт задаётся `ADMIN_PORT` в `.env`) |
| PostgreSQL **с хоста** | `127.0.0.1:5433` → контейнерный `5432` (см. `docker-compose.yml`, проброс только на localhost) |

Внутри сети Compose сервисы подключаются к БД по хосту `postgres` и порту `5432`; строка `DATABASE_URL` для контейнеров подставляется из `docker-compose.yml`, **вручную экспортировать её в контейнерах не нужно**.

### 2.4. Миграции схемы БД

Выполняются **автоматически** при старте контейнера **admin** (`alembic upgrade head`). Отдельно запускать миграции на сервере не требуется, если admin успешно поднялся.

Проверка логов админки:

```bash
docker compose logs -f admin
```

## 2.5. Панель: дашборд, группы, события

- **Дашборд** (`/dashboard`, дубликат на `/`) — плитки **Пользователи**, **Группы**, **События** со счётчиками, блок «Что сделать сейчас» и **Сервис** (строка **Статус:** Включен / Выключен / Рестарт и т.д., дата старта и **uptime** с момента последнего запуска, счётчик ошибок последнего цикла — метрики обновляются **каждую секунду**). Кнопки **Старт / Стоп / Рестарт** в отдельном блоке управляют контейнером бота. **Вход** и **выход** из панели дописываются в файл событий с префиксом `[ADMIN]`. Списки по-прежнему на отдельных страницах.
- **Группы** — комната Matrix группы, таймзона, **статусы** (точные имена статусов Redmine) и **версии** (подстрока в названии версии задачи → дополнительная отправка в комнату группы). Старый URL `/routes/status` перенаправляется сюда.
- **Пользователи** — при редактировании доступен блок **версий** по тому же принципу (личная Matrix-комната пользователя). Глобальная карта **`/routes/version`** (версия → комната для всего бота) по-прежнему в меню настроек.
- **Аккаунты панели** (`/app-users`) — логин, роль, колонка **Создание** (дата появления записи в БД в формате **ДД.ММ.ГГГ ЧЧ:ММ:СС** в **`BOT_TIMEZONE`**), **смена логина** и **сброс пароля** администратором (те же правила формата логина, что при регистрации; при заданном **`ADMIN_LOGINS`** новый логин должен быть в списке).
- **События** (`/events`) — **таблица по файлу лога** (колонки **дата**, **время**, **уровень** вроде `INFO` / `WARNING` / `ADMIN`, **сообщение**): строки сортируются **от новых к старым**. Фильтры **«с даты / по дату»** и выгрузка **`/events/export.csv`** относятся к **этому же файлу** (календарные дни в **`BOT_TIMEZONE`**). Источник: **`ADMIN_EVENTS_LOG_PATH`** или по умолчанию `data/bot.log`. Если файл больше **`ADMIN_EVENTS_LOG_SCAN_BYTES`** (по умолчанию 8 МБ), для страницы читается **хвост**; в UI показывается предупреждение о хвосте, фильтр по датам действует **только по загруженному фрагменту**. Префикс `YYYY-MM-DD` из стандартного логирования в Docker обычно в **UTC**: по умолчанию перевод в **`BOT_TIMEZONE`** (`ADMIN_EVENTS_LOG_PARSE_AS_UTC=1`); если лог уже в локальной зоне — **`ADMIN_EVENTS_LOG_PARSE_AS_UTC=0`**. Операции **Start / Stop / Restart** и **вход/выход** дописывают в этот файл строки с **`[ADMIN]`** (том `data/` у admin с **записью**). Старые URL **`/audit`** и **`/audit/export.csv`** перенаправляют на **`/events`** и **`/events/export.csv`**.

  **Что происходит, когда текущий файл достигает `LOG_MAX_BYTES` (например, 5 МБ):** запись в него прекращается; файл переименовывается в `bot.log.1`; прежний `bot.log.1` сдвигается в `bot.log.2` и т.д.; создаётся **новый пустой** `bot.log`, в него пишутся дальнейшие сообщения. Если архивов больше, чем разрешено **`LOG_BACKUP_COUNT`**, самый старый из цепочки **удаляется**. Раздел «События» читает **текущий** файл (архивы `bot.log.N` в UI не подмешиваются).

  **Журнал аудита панели (отдельный файл):** операции **Docker** над ботом и записи **ADMIN_CRUD** (при любом сценарии вызова `_persist_admin_crud_audit`) дополнительно дописываются в файл с меткой **`[AUDIT]`**. Путь: **`ADMIN_AUDIT_LOG_PATH`** или по умолчанию `data/admin_audit.log`; отключить запись: **`ADMIN_AUDIT_LOG_PATH=-`** (или `none` / `off` / `0` / `false`). На странице «События» этот файл **не показывается** — его смотрят на диске или собирают внешним агентом. Таблица **`bot_ops_audit`** в PostgreSQL по-прежнему заполняется **опционально** (**`ADMIN_AUDIT_CRUD_DB`**, см. `.env.example`) для SQL-отчётов; в UI она не выводится.

  **Строки `[ADMIN]` в файле событий:** всегда **вход**, **выход** и **Docker**. При **`ADMIN_EVENTS_LOG_CRUD=1`** в тот же файл попадают строки **`CRUD …/…`** (пользователи бота, группы, маршруты, «Мои настройки»); секреты в текст не попадают. По умолчанию файловый CRUD **выключен**. Stdout контейнера admin — отдельно (секреты, сброс пароля аккаунтов и т.д.). Матрица каналов: [ADMIN_EVENTS_AND_AUDIT_PLAN.md](ADMIN_EVENTS_AND_AUDIT_PLAN.md).

- Конфигурация пользователей и маршрутов хранится в **PostgreSQL**; после правок в UI перезапустите контейнер **`bot`**, чтобы он подхватил данные.

### Подписка через Matrix (без панели)

Сотрудник пишет боту в **личку** (DM), команда **`!start`**, затем по шагам: API-ключ Redmine (проверяется через Redmine), отдел (имя **группы** из панели), часы и дни, типы уведомлений. Ключ хранится в БД **зашифрованным** (нужен **`APP_MASTER_KEY`** у процесса бота). После успешного завершения список пользователей в боте **обновляется без перезапуска**. Команды **`!change`**, **`!cancel`**, **`!help`**, **`!status`**. Подробности и ограничения: [MATRIX_ONBOARDING_PLAN.md](MATRIX_ONBOARDING_PLAN.md).

## 3. Первый администратор и вход

### 3.1. Одноразовая регистрация (`/setup`)

1. Откройте в браузере: `http://<ваш-хост>:8080/setup` (локально: `http://127.0.0.1:8080/setup`).
2. Страница доступна, пока в БД **ещё нет** ни одного администратора. Задайте логин, пароль и подтверждение по политике сложности (подсказки на форме).
3. После успешной настройки используйте вход: `http://<хост>:8080/login`.

Если `/setup` отвечает, что настройка уже выполнена, значит учётная запись админа уже есть — используйте `/login` или сброс пароля (ниже).

**Редирект с `/setup` сразу на `/login`:** страница `/setup` доступна только пока в таблице `bot_app_users` **нет** ни одного пользователя с ролью `admin`. Если вы «ещё не создавали» админа в этом браузере, но том PostgreSQL (`postgres_data`) уже использовался раньше, запись админа могла остаться с прошлых запусков. Варианты: войти под существующим логином, сбросить пароль скриптом `scripts/reset_admin_password.py`, либо для **чистой** среды — `docker compose down -v` (удалит данные БД) и снова миграции + `/setup`.

### 3.2. Ограничение по логину (`ADMIN_LOGINS`)

В `.env` переменная **`ADMIN_LOGINS`** — список через запятую разрешённых логинов для `/setup` и `/login`. Пустое значение = без ограничения.

### 3.3. Режим «первый вошедший — админ» (опционально)

`ADMIN_BOOTSTRAP_FIRST_ADMIN=1` в `.env` позволяет первому успешно вошедшему по `/login` получить права администратора при определённых условиях в коде. Для production обычно держат `0` и полагаются на `/setup` или заранее созданных пользователей.

## 4. Смена и восстановление пароля **админки** (не Postgres)

1. **Самообслуживания по почте / форме «забыли пароль» в UI нет** — ссылка с входа убрана; `GET /forgot-password` ведёт на страницу входа.
2. Сброс пароля: **другой администратор** в разделе **«Аккаунты панели»** или скрипт `scripts/reset_admin_password.py` при доступе к БД.
3. Страница **`/reset-password?token=…`** остаётся для смены пароля по одноразовому токену, если токен выдан вручную/скриптом (в БД таблица `password_reset_tokens`).

Прямая смена пароля по ссылке из письма в проекте **не используется**.

## 5. После изменений в панели (пользователи, маршруты, секреты)

**Группы:** в БД после миграций по умолчанию есть только служебная группа **UNASSIGNED**. Названия отделов и привязки к Matrix-комнатам каждая организация заводит сама в разделе «Группы» панели.

Бот **подгружает конфигурацию из Postgres при старте** (и в процессе работы — в соответствии с текущей логикой `bot.py`). После существенных изменений в админке **перезапустите сервис bot**:

- через UI панели (если настроен Docker control), или
- вручную: `docker compose restart bot`

Логи бота:

```bash
docker compose logs -f bot
```

## 6. Локальные команды с хоста (Alembic, pytest)

Когда Postgres доступен с машины разработчика (Compose поднят, порт **5433**):

```bash
cd /path/to/matrix_bot_firebeard
source venv/bin/activate   # если используете venv
export DATABASE_URL=postgresql://bot:<ПАРОЛЬ_ИЗ_.env>@127.0.0.1:5433/redmine_matrix
export APP_MASTER_KEY=<32-символьный ключ или как в .env>
pip install -r requirements.txt -r requirements-test.txt
python -m alembic upgrade head
python scripts/check_env_sync.py
python -m pytest tests/ -q --tb=short --ignore=tests/e2e
```

Если `DATABASE_URL` не задан, часть тестов будет пропущена (skip) — это нормально для быстрой проверки без БД.

**Примечание:** пароль в URL должен совпадать с `POSTGRES_PASSWORD` в `.env`. Спецсимволы в пароле — используйте `POSTGRES_PASSWORD_URLENC` в compose; для ручного `export` на хосте URL-кодируйте пароль.

## 7. Смена пароля пользователя **PostgreSQL** (операционная задача)

Не путать с паролем админки.

1. Войти в Postgres с правами суперпользователя (через `docker compose exec postgres psql -U bot -d redmine_matrix` или отдельный admin-аккаунт).
2. Выполнить: `ALTER USER bot WITH PASSWORD 'новый_надёжный_пароль';`
3. Обновить **`POSTGRES_PASSWORD`** и при необходимости **`POSTGRES_PASSWORD_URLENC`** в `.env`.
4. `docker compose up -d` (пересоздание контейнеров с новым окружением).

Если том `postgres_data` уже создан со старым паролем, смена только в `.env` без `ALTER USER` приведёт к ошибкам аутентификации.

## 8. Остановка и резервное копирование

```bash
docker compose down
```

Данные БД в именованном томе `postgres_data` при `docker compose down` **не удаляются**. Резервные копии — стандартные средства PostgreSQL (`pg_dump`) по согласованию с вашей политикой.

## 9. Типичные проблемы

### 9.1. `password authentication failed for user "bot"` (Alembic / pytest с хоста)

Postgres **уже запущен**, но пароль в `DATABASE_URL` **не совпадает** с тем, что записан в данных кластера.

Частые причины:

1. **Том `postgres_data` создан раньше** с другим `POSTGRES_PASSWORD`. PostgreSQL читает пароль суперпользователя **только при первом создании** каталога данных; смена строки в `.env` потом **сама по себе пароль внутри БД не меняет**.
2. В **`.env` другой пароль**, чем вы подставили в `export DATABASE_URL=...` (например, меняли пароль после копирования примера, а в shell оставили `change-me`).

Что сделать:

- Зайти в **каталог репозитория** (не копировать буквально `cd /path/to/matrix_bot_firebeard` из документации — это пример пути).
- Подставить в `DATABASE_URL` **точно** значение `POSTGRES_PASSWORD` из текущего `.env` (порт с хоста — **5433**, см. compose).
- Если в пароле есть **`@ : / # ? %`** и т.п., **нельзя** вставлять его «как есть» в URL. Используйте уже URL-encoded значение **`POSTGRES_PASSWORD_URLENC`** из `.env` (как в `docker-compose.yml`) или соберите DSN с кодированием:

  ```bash
  cd /ваш/путь/matrix_bot_firebeard
  set -a && source .env && set +a
  export DATABASE_URL="$(python3 -c "import os; from urllib.parse import quote_plus; u=os.environ.get('POSTGRES_USER','bot'); p=os.environ.get('POSTGRES_PASSWORD',''); d=os.environ.get('POSTGRES_DB','redmine_matrix'); print(f'postgresql://{quote_plus(u)}:{quote_plus(p)}@127.0.0.1:5433/{d}')")"
  python -m alembic upgrade head
  ```
- Либо выровнять пароль в БД и в `.env`:

  ```bash
  docker compose exec postgres psql -U bot -d redmine_matrix -c "ALTER USER bot WITH PASSWORD 'ваш_новый_пароль';"
  ```

  затем тот же пароль в `POSTGRES_PASSWORD` / `POSTGRES_PASSWORD_URLENC` и `docker compose up -d`.

- Если это **чистый dev** и данные не жалко: остановить compose, удалить том `postgres_data`, снова `docker compose up -d` — кластер создастся заново с паролем из текущего `.env` (**все данные БД пропадут**).

### 9.2. В браузере `ERR_CONNECTION_REFUSED` на `127.0.0.1`

Админка слушает порт **`8080`** по умолчанию (`ADMIN_PORT` в `.env`). Нужен полный URL, например:

`http://127.0.0.1:8080/setup` или `http://127.0.0.1:8080/login`.

Если отказ соединения сохраняется:

```bash
docker compose ps
docker compose logs --tail=80 admin
```

Убедитесь, что контейнер **admin** в состоянии `running`, а не `Restarting` / `Exited` (часто из‑за отсутствия master key, ошибок миграций или прав на файлы).

## 10. Куда не класть секреты

- Не коммитить файл `.env` с реальными паролями.
- Не вшивать пароли БД и API-ключи в исходный код Python — только переменные окружения, файлы секретов или менеджер секретов (см. [secrets-storage.md](secrets-storage.md)).
