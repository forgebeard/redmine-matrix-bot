# Диагностика: бесконечное "Загрузка предпросмотра"

> Obsolete note: документ описывает этап миграции. В текущем UI кнопка ручного обновления предпросмотра удалена, используется автопревью с debounce.

## Симптом

Во вкладке `Уведомления` при переключении в режим редактирования кода предпросмотр застревал в состоянии `Загрузка предпросмотра…` и не переходил в финальное состояние `ok` или `error`.

## Root Cause

- UI имел два разнородных потока: block-editor (`BlockEditor`) и code-only карточки.
- Для части шаблонов рендер управлялся через `BlockEditor`, для части - через отдельные обработчики `tpl-v2-preview`.
- При инициализации/переключении вкладки старые async-ответы preview могли перетирать новое состояние карточки.
- В code-path не было единообразной модели состояний и защиты от гонок (request token), поэтому элемент мог остаться в "loading".

## Исправление

- Полный cutover на единый code-only поток во `static/admin/js/onboarding.js`.
- Для всех шаблонов используется единый endpoint `/api/bot/notification-templates/preview`.
- Введен единый lifecycle preview:
  - `loading`: при отправке запроса;
  - `ok`: успешный рендер HTML;
  - `error`: детализация ошибки рендера/сети без зависания.
- Добавлена защита от гонок (request token): старые ответы игнорируются, если уже отправлен новый запрос.
- Добавлен debounce (400ms) по вводу.

## Результат

- Состояние `Загрузка...` больше не зависает при наборе и переключении.
- Последний введенный код всегда соответствует отображаемому preview.
