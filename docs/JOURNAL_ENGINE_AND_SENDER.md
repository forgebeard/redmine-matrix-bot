# Журнальный движок и путь `send_safe`

## Планировщик

Основной интервальный цикл [`check_all_users`](../src/bot/scheduler.py) вызывает только [`run_journal_tick`](../src/bot/journal_tick.py): обработка назначенных задач через [`journal_handlers`](../src/bot/journal_handlers.py) и именованные шаблоны `tpl_*`.

Ключ **`JOURNAL_ENGINE_ENABLED`** в таблице `cycle_settings` **не читается в текущем Python-коде бота** (нет ветвления по этому значению в `src/`). Если ключ есть в базе, он может использоваться только как операционная заметка или исторический маркер; источником правды о том, какой код выполняется, является именно модуль [`scheduler.py`](../src/bot/scheduler.py).

Ранее существовший цикл **`check_user_issues`** (legacy per-user polling) удалён из репозитория как неисполняемый боевой путь.

## Исключения

Отдельная задача **`check_unassigned_new_issues`** продолжает вызывать **`send_safe`** для новых неназначенных задач. Тело сообщения собирается тем же **`build_matrix_message_content`**, что и для назначенных пользователей: через `tpl_new_issue` и сессию БД.

Итог: визуальный стиль **unassigned new** должен совпадать с настройками тех же `tpl_*`, что и в журнальной системе v2.
